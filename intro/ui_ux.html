Frontend Explanation

The UI is server-rendered first, then upgraded by focused vanilla JS modules.

The templates define structure and placeholders, while JavaScript handles data fetching, optimistic UI updates, theme application, sorting, filtering, tab-specific loading, and assistant streaming. There is no client framework hiding the wiring.

Page Map

The project uses shared includes, then gives each page its own content and behavior instead of one universal mega-template.

Why Sections Are Separated

The code points to practical reasons for the layout rather than abstract design theory.

  • Fast-moving and slow-moving data are split. Metrics are polled globally in `static/js/system_metrics.js`, while storage partitions, system settings, and log-file lists are fetched only when their page or tab needs them.
  • Operational views match the job to be done. The System page is an inspection and control surface, the Network page is a network/firewall workbench, the Users page is a master-detail admin view, and the Logs page is optimized for scanning lines rather than editing data.
  • Shared chrome stays shared. `nav.html`, `controls.html`, `lady_panel.html`, and `context_nav.html` keep navigation, fullscreen controls, and the floating Lady panel consistent across pages.

Live Updates and Event Flow

The frontend uses two layers of eventing because the project has both browser-side fan-out and backend-pushed theme changes.

Local event bus

`static/js/ui_event_bus.js` defines a tiny subscribe/emit API and stores subscribers in a `Map`. `system_metrics.js` emits `metrics:update`, and page-specific modules like `dashboard.js` and `os_metrics.js` subscribe to that stream so one poll can feed multiple widgets.

UI load
  -> system_metrics.js polls /api/system/metrics
  -> normalize payload once
  -> emit("metrics:update", data)
  -> dashboard.js / os_metrics.js render cards and graphs

Backend push for themes

The same `ui_event_bus.js` file also opens a websocket to `/ws/ui`. That path is backend-driven: `theme_service.apply_theme()` persists theme state, publishes a `theme_change` event on `core/event_bus.py`, and the socket listener applies CSS variables on the client.

theme click
  -> POST /api/theme/theme/{name}/apply
  -> config/theme_state.json updated
  -> backend event_bus.publish(...)
  -> /ws/ui pushes theme_change
  -> client applies CSS vars

Loading States That Show Up in Code

  • Skeletons on the System page. `templates/os.html` ships dedicated `.metric-skeleton` placeholders and `static/js/os_metrics.js` removes them once the first real metrics payload lands.
  • Spinner rows on data-heavy pages. Network, users, and logs all initialize their table bodies or content areas with loading rows before async fetches replace them.
  • `data-ui-ready` guards against flash-of-unstyled-content. Multiple templates set `data-ui-ready="false"` before the design and theme scripts run, then the JS initialization path flips the UI into the ready state.

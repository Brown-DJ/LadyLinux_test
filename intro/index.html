Two Major Request Paths

The codebase exposes one regular dashboard/data path and one assistant path. They share service modules, but they do not start in the same place.

Page modules and data panels

templates + static/js
Jinja page shell Shared includes such as `nav.html`, `controls.html`, and `lady_panel.html` wrap module-specific page bodies.
->
Vanilla JS fetch layer Files like `system_metrics.js`, `main.js`, and the inline scripts in `os.html`, `network.html`, `users.html`, and `logs.html` call backend endpoints.
->
FastAPI route modules `api_layer/routes/*` contains thin endpoint wrappers grouped by area.
->
Service layer `api_layer/services/*` runs psutil sampling, file reads, theme persistence, or command execution through `run_command()`.

Prompt transport and decision logic

api_layer/app.py + core/*
Prompt entry The current frontend posts to `/api/prompt` or `/api/prompt/stream`.
->
Command kernel first `run_command_kernel()` checks deterministic tool-like prompts before retrieval or chat.
->
Prompt router `classify_prompt()` chooses `system`, `rag`, or `chat`.
->
Tool, RAG, or chat handler The request is executed through the tool router, retrieval pipeline, or plain LLM chat path.

Observed Building Blocks

These cards summarize what the scan found without inventing hidden layers.

Frontend shell

Pages are server-rendered with Jinja2 templates, but most live behavior is delegated to vanilla JavaScript modules and page-local scripts.

`templates/index.html` `templates/os.html` `static/js/main.js`

Shared live updates

A shared polling engine reads `/api/system/metrics`, normalizes the payload once, then emits `metrics:update` so multiple pages can render from the same stream.

`static/js/system_metrics.js` `static/js/ui_event_bus.js`

Route composition

Most endpoints live in modular routers under `api_layer/routes`, while the assistant transport and some backward-compatible aliases remain in `api_layer/app.py`.

`api_layer/routes/*` `api_layer/app.py`

System-facing services

The service layer mixes psutil, file reads, theme-state persistence, subprocess calls, and in one case a websocket-aware event bus for UI theme pushes.

`api_layer/services/*` `core/event_bus.py`

Why This Architecture Reads Clearly

  • Feature pages stay human-sized. The template layer uses shared includes for navigation and controls, then hands each page its own JS hooks and area-specific sections.
  • The assistant path is explicit. Instead of hiding prompt logic inside one giant service, the code separates command matching, tool routing, retrieval, and chat response generation.
  • System work is centralized. Subprocess execution is funneled through `api_layer/utils/command_runner.py`, so allowed binaries and result formatting stay in one place.
